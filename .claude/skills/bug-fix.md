# Bug Fix — 缺陷修复

## 触发条件

- "修这个 bug"
- "报错了，帮我看看"
- "XX 功能不正常"
- 用户贴了一段错误日志或异常堆栈

## 核心原则

> **先诊断，再下药。必须写回归测试，防止复发。**

---

## 执行流程

### Phase 1: 理解问题

1. 阅读用户提供的错误信息 / 日志 / 堆栈
2. 定位问题代码（搜索相关文件，理解上下文）
3. 判断影响范围（这个 bug 影响哪些功能？）

### Phase 2: 诊断（先红）

4. **先写一个复现该 bug 的测试** — 这个测试当前应该失败（红）
5. 确认测试确实复现了问题

```
// 例如：shouldThrowException_whenEmailDuplicate
@Test
void shouldThrowException_whenEmailIsDuplicate() { ... }  // ← 红
```

### Phase 3: 修复（后绿）

6. 修改代码，修复根因而非表象
7. 运行测试，确认新测试通过（绿）
8. 运行全量测试，确认无回归
   ```bash
   ./mvnw test       # 后端
   npm run test      # 前端
   ```

### Phase 4: 记录

9. 在 worklog 中记录：
   - 根因是什么
   - 修复了什么
   - 如何防止复发

     ```markdown
     ### HH:MM — Bug Fix: xxx

     - **类型：** Bug修复
     - **根因：** xxx
     - **修复：** xxx
     - **测试：** 添加 shouldXxx 回归测试
     ```

---

## Checklist

- [ ] 理解根因，不是修表象
- [ ] 先写了复现测试（先红）
- [ ] 修复后测试通过（后绿）
- [ ] 全量测试无回归
- [ ] 新增回归测试
- [ ] 不影响其他功能

## 禁止行为

- ❌ 不写测试就改代码
- ❌ 修表象不修根因（如 catch 吞异常）
- ❌ 修复后不跑全量测试
