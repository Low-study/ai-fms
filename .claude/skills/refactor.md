# Refactor — 代码重构

## 触发条件

- "重构这段代码"
- "这个类太大了，拆分一下"
- "这个方法的逻辑需要整理"
- "优化这段代码结构"

## 核心原则

> **外部行为不变，内部结构优化。用测试做安全网。**

---

## 执行流程

### Phase 1: 理解现状

1. 阅读目标代码，理解其职责和依赖关系
2. 确认现有的测试覆盖情况
3. 如果测试覆盖不足 → **先补测试**，再重构
4. 确定重构目标：
   - 拆大类？ → 按职责分离
   - 减重复？ → 抽取公共方法/类
   - 提可读？ → 重命名 + 提取方法

### Phase 2: 设计

5. 给出重构方案：
   - 改名？拆类？提取方法？
   - 影响哪些调用方？
6. 确认方案后再动手

### Phase 3: 小步执行

```
每次只改一步，改完就跑测试：

7.  提取方法  → 跑测试
8.  重命名    → 跑测试
9.  移动类    → 跑测试
10. 删除冗余 → 跑测试
```

### Phase 4: 验证

11. `./mvnw test` 全绿
12. `npm run test` 全绿（如涉及前端）
13. 确认无新增警告

### Phase 5: 记录

14. 更新 worklog，记录重构了什么、为什么

---

## 常见重构手法

| 场景 | 手法 |
|------|------|
| 方法 > 50 行 | Extract Method |
| 类 > 500 行 | Extract Class |
| 重复代码 | Pull Up / Extract Method |
| 参数 > 5 个 | Introduce Parameter Object |
| 魔法数字 | Replace Magic Number with Constant |
| 嵌套 > 3 层 | Replace Nested Conditional with Guard Clauses |
| 命名不清 | Rename Method / Variable |

## Checklist

- [ ] 重构前测试全绿
- [ ] 测试覆盖不足时先补测试
- [ ] 每次只改一步，改完跑测试
- [ ] 外部API行为不变
- [ ] 全量测试通过
- [ ] 无新增技术债务

## 禁止行为

- ❌ 重构 + 改功能一起做
- ❌ 跳过测试直接重构
- ❌ 大爆炸式重构（一次改几十个文件）
